## 🔴 CRITICAL ISSUES

### 1. In-Memory Storage Used in Production Code
**File:** `server/storage.ts`  
**Impact:** Data loss on server restart, not scalable

The storage layer exports a `MemStorage` class that stores all data in JavaScript Maps:

```typescript
export class MemStorage implements IStorage {
  private users: Map<number, User>;
  private personas: Map<number, UserPersona>;
  // ... all in-memory
}

export const storage = new MemStorage();
```

**Problem:** 
- All user data, conversations, personas are lost on every server restart
- Database schema exists (`schema.ts`) and db is configured (`db.ts`) but NEVER USED
- The `db` export in `db.ts` is imported nowhere in the codebase

**Fix Required:**
- Implement `DatabaseStorage` class that uses `db` from `db.ts`
- Run Drizzle migrations to create tables
- Replace `MemStorage` instantiation with `DatabaseStorage`

### 2. getUserStats() Not User-Scoped
**File:** `server/storage.ts` (lines 378-469)  
**Impact:** Security vulnerability - users see aggregate stats, not their own

```typescript
async getUserStats(): Promise<{...}> {
  const conversations = Array.from(this.conversations.values()); // ALL conversations
  const messages = Array.from(this.messages.values());           // ALL messages
  // ...
}
```

**Problem:** This returns stats for ALL users combined, not the authenticated user.

**Fix:**
```typescript
async getUserStats(userId: number): Promise<{...}> {
  const userConversations = Array.from(this.conversations.values())
    .filter(c => c.userId === userId);
  // ...
}
```

### 3. Missing Conversation-User Association
**File:** `shared/schema.ts` (line 59-64)  
**Impact:** Cannot determine which user owns a conversation

```typescript
export const conversations = pgTable("conversations", {
  id: serial("id").primaryKey(),
  personaId: integer("persona_id"),  // Only persona, no userId!
  title: text("title").default("New Conversation"),
  createdAt: timestamp("created_at").default(sql`CURRENT_TIMESTAMP`).notNull(),
});
```

**Problem:** Conversations link to `personaId` but not directly to `userId`. While users can be traced through personas, this indirect relationship:
- Makes queries inefficient
- Causes issues if a user has multiple personas or no persona
- Requires joins for simple user-conversation checks

**Fix:** Add `userId` column:
```typescript
userId: integer("user_id").references(() => users.id, { onDelete: "cascade" }),
```

## 🟠 HIGH PRIORITY ISSUES

### 5. routes.ts is 1430 Lines - Needs Splitting
**File:** `server/routes.ts`  
**Impact:** Maintainability nightmare, hard to test

One file handles: auth, onboarding, conversations, messages, chat streaming, Bible API, devotionals, personas, trust tracking, shame detection, mode transitions.

**Fix:** Split into route modules:
```
server/routes/
  auth.ts
  conversations.ts
  chat.ts
  bible.ts
  devotionals.ts
  persona.ts
```

### 6. Dynamic Imports in Route Handlers
**File:** `server/routes.ts` (lines 1192, 1234, 1260, 1298, 1376, 1418)  
**Impact:** Performance hit on every request

```typescript
app.post("/api/persona/trust", async (req, res) => {
  // Inside the handler!
  const { trustTrackingService } = await import("./services/trustTrackingService");
  // ...
});
```

**Problem:** Dynamic imports add latency to every API call. These services should be imported at the top of the file.

**Fix:** Move to top-level imports:
```typescript
import { trustTrackingService } from "./services/trustTrackingService";
import { modeTransitionService } from "./services/modeTransitionService";
import { shameLoggingService } from "./services/shameLoggingService";
```

### 7. Type Safety Bypassed with `as any`
**Files:** `server/routes.ts` (lines 160, 233, 236)  
**Impact:** Runtime errors, TypeScript protection disabled

```typescript
const userId = (req.session as any)?.userId;  // Line 160
const personaAssignment = assignPersona(onboardingData as any);  // Line 236
```

**Fix:** Properly type the session:
```typescript
declare module 'express-session' {
  interface SessionData {
    userId?: number;
  }
}
```

### 8. Chat Streaming - No Error Recovery After Partial Response
**File:** `server/services/hybridAIClient.ts` (lines 102-121)  
**Impact:** User sees partial AI response with no completion

```typescript
if (hasYieldedContent) {
  console.warn("[HybridAI] Claude failed mid-stream after yielding content...");
  // No recovery! User sees incomplete message
}
```

**Problem:** If Claude fails mid-stream, the user sees a truncated response with no indication it's incomplete.

**Fix:** 
1. Append an error indicator to the stream
2. Or buffer responses and only yield complete ones
3. Or mark the message as incomplete in the database

### 9. localStorage for Conversation ID is Insecure
**File:** `client/src/pages/Chat.tsx` (line 169, 192)  
**Impact:** Cross-user data access, XSS vulnerability

```typescript
localStorage.setItem("soulguide_conversation_id", String(data.id));
// ...
const savedConvId = localStorage.getItem("soulguide_conversation_id");
```

**Problems:**

- Always validate conversation ownership server-side before loading

### 10. No Rate Limiting on AI Endpoints
**File:** `server/routes.ts`

No rate limiting on:
- `/api/conversations/:id/message` (AI chat)
- `/api/devotionals/generate` (AI generation)

```typescript
const chatLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: 10, // 10 messages per minute
});

app.post("/api/conversations/:id/message", chatLimiter, async (req, res) => {
```

## 🟡 MEDIUM PRIORITY ISSUES

### 13. Shame Detection Relies on Simple Regex
**File:** `server/services/gracePersonaSystem.ts` (lines 20-68)  

Regex patterns like `/i('m| am) (so )?(bad|broken|terrible)/i` are:
- Easy to bypass ("I feel worthless" vs "I am worthless")
- May false-positive on literary/educational discussions
- No context awareness

**Recommendation:** Consider using AI for sentiment analysis on sensitive detection, or implement a confidence scoring system.

```typescript
async getAllConversations(): Promise<Conversation[]> {
```
